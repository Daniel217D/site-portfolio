# FSD Rules

## Назначение

Этот skill фиксирует рабочие правила использования FSD в проекте.

Цель - помогать агенту принимать одинаковые архитектурные решения при добавлении новых страниц, виджетов и интеграций.

## Основной принцип

Архитектура Nuxt адаптируется под FSD-подход, а не наоборот.

Это означает:

* сначала определяется слой ответственности;
* затем выбирается способ интеграции с Nuxt;
* framework-ограничения не должны размывать границы доменных слоёв без причины.

## Слои

`pages`

* собирает страницу из виджетов и сущностей;
* не должен хранить низкоуровневые API-утилиты;
* содержит route-level композицию.

`widgets`

* содержит крупные интерфейсные блоки;
* может собирать несколько features и entities;
* не должен превращаться в склад случайных компонентов.

`features`

* содержит пользовательские сценарии и прикладную интерактивность;
* изолирует логику взаимодействия с demo API;
* должен иметь чёткую функциональную границу.

`entities`

* описывает сущности предметной области;
* в текущем проекте прежде всего это `project`;
* не должен зависеть от page-level композиции.

`shared`

* содержит переиспользуемые API-клиенты, конфиг, утилиты и базовый UI;
* не должен знать о конкретных страницах проекта;
* используется как инфраструктурный фундамент.

## Практические правила

* Не поднимать код выше по слоям "на всякий случай".
* Не опускать доменную логику в `shared`, если она относится к конкретной фиче.
* Если компонент нужен только одной странице, не делать его глобальным заранее.
* Если логика demo уникальна для проекта, держать её рядом с соответствующей feature/widget-композицией.
* Если появляются повторяющиеся паттерны, выносить их после второго-третьего подтверждённого использования.

## Когда обновлять спецификацию

Обновляй `specs/architecture.md`, если:

* появляется новый слой или обязательный подслой;
* меняется правило размещения кода;
* добавляется новый устойчивый архитектурный паттерн;
* меняются границы между `widgets`, `features`, `entities`, `shared`.
