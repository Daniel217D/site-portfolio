# Demo Component

## Назначение

Этот skill описывает правила создания demo-компонентов для интеграции с внешними ML API.

Цель - сделать интерактивные демо изолированными, проверяемыми и безопасными для статического фронтенда.

## Базовые правила

* Каждое demo может иметь полностью уникальную реализацию под конкретный проект.
* Demo работает с внешним API на клиенте.
* Production Node.js runtime не предполагается.
* Публичные переменные окружения допускаются по текущему замыслу.

## Переменные окружения

Используются:

```env
NUXT_PUBLIC_DEMO_API_URL=
NUXT_PUBLIC_DEMO_API_KEY=
```

Правила:

* ключ считается публичным и не должен трактоваться как секретный backend credential;
* URL и key должны читаться через публичную конфигурацию Nuxt;
* не хардкодить endpoint и ключ в компоненте.

## Архитектурные правила

* UI demo размещается в `widgets/project-demo` или в проектно-специфичной композиции рядом.
* Логику запроса выносить в `features/demo-api-request` и/или `shared/api`, если она становится переиспользуемой.
* Компонент должен отделять форму, состояние запроса и отображение результата.
* Ошибки API должны быть показаны пользователю в контролируемом виде.

## Минимальный сценарий demo

1. Пользователь вводит данные.
2. Компонент валидирует минимальные условия отправки.
3. Выполняется клиентский запрос к внешнему API.
4. Показывается состояние загрузки.
5. Показывается результат или ошибка.

## Тестирование

Для demo-компонентов обязательны unit-тесты, если в них есть нетривиальная логика:

* валидация ввода;
* обработка ошибок;
* трансформация ответа;
* переключение состояний загрузки и результата.

## Нельзя

* коммитить реальные секреты в `.env`;
* смешивать UI формы и низкоуровневый HTTP-клиент в большой монолитный файл;
* завязывать demo на недоступный в production серверный runtime;
* использовать непроверяемую магию без явного состояния ошибок и загрузки.
