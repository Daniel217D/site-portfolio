# Project Structure

## Назначение

Этот skill описывает базовую структуру проекта и правила размещения новых файлов.

Цель - поддерживать единообразную организацию кода и не размывать FSD-слои по мере роста проекта.

## Базовая структура

Проект организуется внутри `src/`:

```text
src/
├── app/
├── pages/
├── widgets/
├── features/
├── entities/
└── shared/
```

Предварительная структура проекта:

```text
src/
├── app/
├── pages/
│   ├── home/
│   └── project/
├── widgets/
│   ├── project-page/
│   ├── project-presentation/
│   └── project-demo/
├── features/
│   └── demo-api-request/
├── entities/
│   └── project/
└── shared/
    ├── api/
    ├── config/
    ├── lib/
    └── ui/
```

## Правила размещения

* Новые файлы сначала соотносятся со слоем FSD, а уже потом с технической реализацией Nuxt.
* Глобальная конфигурация, провайдеры и app-level wiring размещаются в `src/app`.
* Маршрутные сценарии и page-level композиция размещаются в `src/pages`.
* Крупные составные блоки интерфейса размещаются в `src/widgets`.
* Пользовательские действия и прикладная логика размещаются в `src/features`.
* Бизнес-сущности и их минимальная модель размещаются в `src/entities`.
* Переиспользуемая инфраструктура и базовый UI размещаются в `src/shared`.

## Правила изменений

* Не добавлять новые top-level каталоги без отдельной необходимости и обновления спецификаций.
* Не переносить код между слоями без объяснимой причины.
* Не смешивать page-композицию, feature-логику и shared-утилиты в одном месте.
* Если структура проекта меняется, сначала обновить спецификацию архитектуры.

## Чеклист перед добавлением файла

1. Определить, это `page`, `widget`, `feature`, `entity` или `shared`.
2. Проверить, не существует ли уже подходящего места в текущей структуре.
3. Выбрать имя каталога по смыслу домена, а не по технической детали.
4. Добавить файл в минимально достаточный слой.
5. Обновить спецификацию, если изменение влияет на архитектурные правила.
