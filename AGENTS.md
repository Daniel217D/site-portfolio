# AGENTS

Этот файл задаёт базовую точку входа для агента в проект.

## Общий принцип

Перед реализацией агент сначала определяет тип задачи, затем сверяется с соответствующими спецификациями в `specs/` и рабочими инструкциями в `.codex/skills/`.

Спецификации являются источником требований.

Skills являются источником прикладных правил выполнения.

## Откуда брать требования

Для общего контекста проекта:

* `specs/_project.md`
* `specs/project.md`

Для архитектуры и структуры:

* `specs/architecture.md`
* `.codex/skills/project-structure.md`
* `.codex/skills/fsd-rules.md`

Для маршрутов и страниц:

* `specs/routing.md`
* `specs/projects.md`
* `.codex/skills/nuxt-page.md`

Для demo-интеграций:

* `specs/demo-api.md`
* `.codex/skills/demo-component.md`

Для презентаций:

* `specs/architecture.md`
* `.codex/skills/reveal-presentation.md`

Для проверок качества:

* `specs/testing.md`
* `.codex/skills/testing.md`

Для Docker и CI/CD:

* `specs/ci-cd.md`
* `.codex/skills/ci-cd.md`

Для работы с Git:

* `specs/git-workflow.md`
* `.codex/skills/git-workflow.md`

## Порядок работы агента

1. Определить, какие спецификации относятся к задаче.
2. Определить, какие skills задают правила реализации.
3. При изменении поведения сначала обновить нужную спецификацию.
4. Затем выполнять реализацию, тесты и Git-действия по соответствующим skills.

## Если задача затрагивает несколько областей

Агент должен использовать все релевантные спецификации и skills вместе, не ограничиваясь одним файлом.

Если между реализацией и спецификацией есть расхождение, приоритет у спецификации.
