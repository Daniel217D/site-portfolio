# Git Workflow Specification

## Статус

Draft v0.1

---

# Назначение

Этот документ описывает правила работы с Git для проекта.

Цель - сделать изменения понятными, проверяемыми и удобными для работы с AI-агентами.

---

# Основные правила

* Не работать напрямую в `main`.
* Каждая задача выполняется в отдельной ветке.
* Один PR должен решать одну логическую задачу.
* Перед коммитом должны проходить проверки проекта.
* Коммиты должны быть осмысленными и небольшими.
* Агент не должен смешивать несвязанные изменения.

---

# Ветки

Базовая ветка проекта:

```text
main
```

Перед созданием рабочей ветки необходимо перейти на актуальную `main`.

Рабочие ветки создаются от `main`, если пользователь явно не указал другую базовую ветку.

Формат веток:

```text
<type>/<short-description>
```

Примеры:

```text
feature/project-pages
feature/reveal-integration
feature/demo-sentiment-form
fix/project-routing
chore/docker-dev-env
docs/update-specs
test/demo-components
```

Допустимые типы:

```text
feature  новая функциональность
fix      исправление ошибки
docs     документация или спецификации
test     тесты
chore    инфраструктура, зависимости, настройки
refactor изменение структуры без изменения поведения
```

---

# Коммиты

Формат коммитов:

```text
<type>: <short summary>
```

Примеры:

```text
docs: add initial project specification
feature: add project pages routing
feature: add reveal presentation widget
fix: handle missing project page
test: add demo form unit tests
chore: add docker development setup
```

---

# Правила коммитов

Коммит должен быть атомарным.

Хороший коммит:

* описывает одно изменение;
* не содержит случайных правок;
* не смешивает код, форматирование и документацию без причины;
* может быть объяснён одной фразой.

Плохие коммиты:

```text
update
fix stuff
wip
changes
final
```

---

# Spec-Driven правило

Если изменение затрагивает поведение проекта, архитектуру, маршруты, API, тесты или CI/CD, сначала обновляется соответствующая спецификация.

Порядок работы:

1. Обновить спецификацию.
2. Реализовать код.
3. Добавить или обновить тесты.
4. Запустить проверки.
5. Сделать коммит.

---

# Проверки перед коммитом

Перед коммитом необходимо выполнить:

```bash
npm run typecheck
npm run lint
npm run test:unit
npm run generate
```

Можно использовать общую команду:

```bash
npm run check
```

---

# Pull Request

PR должен содержать:

* краткое описание задачи;
* список основных изменений;
* список обновлённых спецификаций;
* результат проверок;
* известные ограничения, если есть.

Шаблон:

```md
## Summary

-

## Specs updated

-

## Checks

- [ ] npm run typecheck
- [ ] npm run lint
- [ ] npm run test:unit
- [ ] npm run generate

## Notes

-
```

# Правила для AI-агентов

AI-агент должен:

* проверить текущую ветку перед началом работы;
* не работать в `main`, если это не разрешено явно;
* создать ветку под задачу;
* не переписывать несвязанные файлы;
* не делать массовое форматирование без отдельной задачи;
* не коммитить секреты, `.env`, ключи и токены;
* обновлять спецификации до реализации;
* писать понятные сообщения коммитов;
* показывать пользователю итоговый diff или краткое summary изменений.

AI-агент не должен:

* выполнять `git push --force` без явного разрешения;
* удалять ветки без разрешения;
* менять историю Git без разрешения;
* коммитить временные файлы;
* коммитить `.env`;
* объединять несколько независимых задач в один коммит.

---

# Работа с секретами

Файлы с секретами не коммитятся.

Запрещено коммитить:

```text
.env
.env.local
.env.*.local
```

Для примера используется:

```text
.env.example
```

# Рекомендуемый порядок работы агента

```bash
git status
git checkout main
git checkout -b feature/task-name

# update specs
# implement changes
# add tests

npm run check

git status
git diff
git add <files>
git commit -m "feature: short description"
```

---

# Definition of Done для Git

Задача готова, если:

* изменения находятся в отдельной ветке;
* коммиты атомарные;
* спецификации обновлены;
* проверки проходят;
* нет случайных файлов в diff;
* нет секретов;
* PR можно понять без просмотра всей истории переписки.
